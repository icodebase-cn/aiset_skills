# 评分标准与反馈模板

## 评分维度

| 维度 | 权重（实习） | 权重（应届） | 权重（社招） |
|------|-------------|-------------|-------------|
| AI 技术深度 | 25% | 30% | 35% |
| 项目经验 | 25% | 30% | 35% |
| 思维逻辑 | 25% | 20% | 15% |
| 表达能力 | 15% | 10% | 8% |
| AI 工具使用 | 10% | 10% | 7% |

> 社招权重偏向 AI 技术深度和项目经验（生产级）；实习/应届权重偏向基础和学习潜力。

## 评分等级参考

### AI 技术深度（30%/25%/35%）

| 等级 | 分数段 | 实习表现 | 应届表现 | 社招表现 |
|------|--------|---------|---------|---------|
| 优秀 | 9-10 | — | 能讲清 LLM 底层原理，有系统性理解 | 能讲清 Agent/RAG 源码层面，深入理解 ReAct、Context Engineering |
| 良好 | 7-8 | 对 Token/上下文/Temperature 基本了解正确 | 对 Agent/RAG 核心原理理解正确，能回答"为什么" | 对技术原理有深入理解，能结合生产场景讲解 |
| 一般 | 5-6 | 知道基本概念，能说出一二 | 知道基本概念，停留在使用层面 | 知道基本原理，但无法深入展开 |
| 不足 | 3-4 | 概念模糊，无法准确描述 | 概念模糊，无法准确描述 | 概念模糊，无法准确描述 |
| 较差 | 1-2 | 完全不了解 | 完全不了解 | 完全不了解 |

### 项目经验（25%/30%/35%）

| 等级 | 分数段 | 实习表现 | 应届表现 | 社招表现 |
|------|--------|---------|---------|---------|
| 优秀 | 9-10 | — | 有 RAG/Agent 相关的项目，能说清技术难点 | 真实生产项目，能说清架构选型权衡和线上问题 |
| 良好 | 7-8 | 有完整的课程项目，能描述实现 | 有一定规模项目，能描述技术难点 | 有业务项目，能说清自己的贡献和技术决策 |
| 一般 | 5-6 | 有项目但描述简单 | 项目较小，能说明自己的贡献 | 项目描述偏平，缺乏深度 |
| 不足 | 3-4 | 项目说不清楚 | 项目说不清楚 | 项目说不清楚 |
| 较差 | 1-2 | 无项目经验 | 无真实项目 | 无真实项目 |

### 思维逻辑（25%/20%/15%）

| 等级 | 分数段 | 表现 |
|------|--------|------|
| 优秀 | 9-10 | 思路清晰，能层层拆解问题，给出完整分析路径 |
| 良好 | 7-8 | 逻辑连贯，能有条理地表达观点 |
| 一般 | 5-6 | 基本能回答，但思路有时跳跃 |
| 不足 | 3-4 | 思维混乱，前后矛盾 |
| 较差 | 1-2 | 无法形成有效回答 |

### 表达能力（15%/10%/8%）

| 等级 | 分数段 | 表现 |
|------|--------|------|
| 优秀 | 9-10 | 表达流畅，专业术语准确，能用类比辅助解释 |
| 良好 | 7-8 | 表达清晰，术语基本准确 |
| 一般 | 5-6 | 能表达基本意思，但有时词不达意 |
| 不足 | 3-4 | 表达困难，需要多次追问才能理解 |
| 较差 | 1-2 | 无法有效沟通 |

### AI 工具使用（10%/10%/7%）

| 等级 | 分数段 | 实习表现 | 应届表现 | 社招表现 |
|------|--------|---------|---------|---------|
| 优秀 | 9-10 | 了解多种 AI 编程工具，有高级使用技巧 | 有实际 AI 编程经验，能说清使用场景和局限 | AI 深度融入工作流，持续跟进新工具 |
| 良好 | 7-8 | 知道 Copilot 等工具，会基本使用 | 了解 AI 编程工具，有一定使用经验 | 有实际应用经验，能说清应用场景和局限 |
| 一般 | 5-6 | 了解 AI 工具但使用较少 | 了解 AI 工具但使用较少 | 了解 AI 工具但应用较少 |
| 不足 | 3-4 | 对 AI 工具几乎不了解 | 对 AI 工具几乎不了解 | 对 AI 工具几乎不了解 |

```
综合评分 = AI技术深度×权重 + 项目经验×权重 + 思维逻辑×权重 + 表达能力×权重 + AI工具使用×权重
```

**最终分数保留 1 位小数**

## 各风格反馈措辞示例

同一份面试结果，六种风格会给出不同的反馈措辞。以下是应届生同一场面试后的反馈对比：

### 严厉拷打型

```
综合评分 5.5/10。

AI 技术深度 5.0：RAG 基本流程能说清楚，但问到向量索引算法（HNSW/IVF）就只停在"向量检索"三个字，
没有深入到索引结构的权衡。不足。

项目经验 5.5：项目规模偏小，QPS、数据量都说不出来，线上故障经验为零。

思维逻辑 6.0：还行，基本能讲清自己做了什么。

表达能力 6.0：基本合格。

AI 工具使用 5.0：Copilot 在用，但说不出 deeper 的用法。

你现在出去面大概率挂。主要问题是项目深度不够 + RAG 召回优化理解浮于表面。建议：
1. 把 HNSW 原理搞透，能说清分层图结构和搜索权衡
2. 整理一个项目中的性能数据，哪怕是召回率/延迟数据也好
3. 上面两条没做到之前别投了。
```

### 温和鼓励型

```
综合评分 5.5/10，整体表现中规中矩，有一些亮点也有一些需要提升的地方。

AI 技术深度 5.0：RAG 基础还是比较扎实的，分块和向量检索原理能讲清楚，继续保持。
如果能把索引算法再深入一些就更好了。

项目经验 5.5：你有一个完整的项目经历，这在应届生中是不错的。
建议接下来可以更深入地了解项目的技术细节，特别是召回优化部分。

思维逻辑 6.0：你的逻辑思维很不错，说话有条理，这点在面试中很重要。

表达能力 6.0：表达清晰，能把自己的想法准确传达出来，继续保持。

AI 工具使用 5.0：看得出来你有在用 AI 工具辅助学习，这是个好的开始。

建议下一步：1. 深入学习 HNSW 原理；2. 整理项目中的召回率和延迟数据。
总体来说基础不错，继续加油！
```

### 专业高效型

```
综合评分 5.5/10。

【结论】AI 技术深度基本合格，项目深度待加强，思维逻辑良好。

AI 技术深度 5.0/10 — RAG 5分，向量索引 4分（HNSW 不熟）
项目经验 5.5/10 — 有项目经历，规模偏小，缺乏数据支撑
思维逻辑 6.0/10 — 逻辑清晰，表达有条理
表达能力 6.0/10 — 准确，无废话
AI 工具使用 5.0/10 — AI 工具使用较浅

【核心短板】项目规模偏小，缺少生产级性能数据；RAG 召回优化深度不足。

【下一步建议】
  1. 阅读 HNSW 论文或源码（重点：分层图结构和搜索权衡）
  2. 用真实数据量化项目性能指标（召回率、延迟）
  3. 系统学习 MCP 协议，准备 2-3 个完整场景设计案例
```

### 深挖学术型

```
综合评分 5.5/10。

你的思维路径比较清晰，但对一些核心机制的理解还停留在"描述性"阶段，
缺乏"分析性"视角。这是应届生常见的问题，需要从"知道是什么"升级到"理解为什么"。

AI 技术深度 5.0：你描述了 RAG 的流程，但没有进一步分析为什么分块策略影响召回质量、
为什么 HNSW 选择分层图而不是平面图。这些问题没有标准答案，但能考察你对算法权衡的理解。

项目经验 5.5：项目完整，但缺少对技术选型的深度复盘。

思维逻辑 6.0：论证路径基本清晰，但部分环节跳跃。

AI 工具使用 5.0：AI 编程工具使用案例比较浅，建议了解 Copilot 的底层原理和限制。

建议下一步：阅读《ReAct》论文和《Dense Passage Retrieval》论文，
从设计者视角理解 RAG 和推理范式的选择逻辑。
```

### 工程实践型

```
综合评分 5.5/10。

你的优势是有完整的项目经历，这是很多应届生缺乏的。但项目的技术深度不够。

AI 技术深度 5.0：能讲清楚基本原理，但结合生产场景就卡住了。
比如你说用到了 RAG，但我问你线上有没有遇到过召回率低的问题，
你说"应该没有吧"——这就是没有复盘的表现。

项目经验 5.5：项目在跑，这很好。但我需要数据：召回率多少？延迟 P99 是多少？
慢查询是哪几个？怎么排查的？这些说不出来，在生产环境会很被动。

思维逻辑 6.0：基本逻辑是清晰的。

建议下一步：
1. 回顾项目的三个核心检索场景，测出召回率和延迟数据
2. 给自己项目做一次召回质量评估，整理成文档
3. 去了解一下生产环境的监控体系（LangSmith/Weave）
```

### 综合平衡型

```
综合评分 5.5/10。

今天的面试表现中规中矩，有一些做得不错的地方，也有明显的提升空间。

AI 技术深度 5.0：RAG 基础是达标的，分块、向量检索都能讲清楚。
不过 HNSW 索引是薄弱点，建议重点看一下。

项目经验 5.5：你有实际参与项目的经历，这是加分项。
不过项目的技术细节还需要更深入，比如能说出具体的召回率数据和技术选型的权衡。

思维逻辑 6.0：思路比较清晰，能有条理地表达自己的观点。

表达能力 6.0：表达流畅，没有太多废话，这点很好。

AI 工具使用 5.0：AI 工具在用，建议深入了解一些高级用法。

下一步建议：
1. 深入 HNSW 原理，重点理解分层图搜索的设计权衡
2. 整理项目中的性能指标和召回质量问题
3. 整体学习路线推荐：LangChain 文档 + RAG 经典论文

今天的面试是一次很好的练习，继续准备，下次会更好。
```

## 面试进行中提示（不透露分数，只给引导）

### 严厉拷打型

| 场景 | 话术 |
|------|------|
| 回答模糊 | "不够。说到哪算哪？重来。" / "你说的这是背的，不是理解的。" |
| 遇到困难 | "这都不知道？那你做这个项目的底气在哪？" |
| 回答较好 | "行了，这个点过了。" / "这种程度还不够，继续。" |
| 压力测试 | "不够。如果线上 RAG 召回率突然腰斩，你打算怎么排查？" |

### 温和鼓励型

| 场景 | 话术 |
|------|------|
| 回答模糊 | "我理解你的意思。能再具体说说吗？比如……" |
| 遇到困难 | "没关系，这个问题确实有点偏。我给你一点提示……" |
| 回答较好 | "很好！这个问题你理解得很到位。能再补充一下吗？" |
| 压力测试 | "好，那我们来想想另一种情况……"（不说"错了"，而是说"那另一种思路呢？"）|

### 专业高效型

| 场景 | 话术 |
|------|------|
| 回答模糊 | "这个回答没有覆盖核心点。我们换一个问法……" |
| 遇到困难 | "好，这个点跳过。基础部分继续。"（不等，不追问）|
| 回答较好 | "好，这个点清楚了。项目深挖继续。" |
| 压力测试 | "我们来讨论一个边界情况：如果……你会怎么处理？" |

### 深挖学术型

| 场景 | 话术 |
|------|------|
| 回答模糊 | "我不太清楚你的论证路径。能先告诉我，你为什么这么认为吗？" |
| 遇到困难 | "这个问题没有标准答案。我更想知道你基于什么假设得出这个结论？" |
| 回答较好 | "好，那我们来延伸一下——这个问题在《ReAct》论文里是怎么论述的？" |
| 压力测试 | "好，你给了方案 A，那有没有人提出过方案 B？他们的核心分歧在哪里？" |

### 工程实践型

| 场景 | 话术 |
|------|------|
| 回答模糊 | "你们实际上线的时候怎么部署的？用的什么向量数据库？" |
| 遇到困难 | "这个问题我在生产环境遇到过，大概是这么个情况……你想想你会怎么排查？" |
| 回答较好 | "好，那你们有没有遇到过向量检索召回了一批不相关文档的问题？怎么处理的？" |
| 压力测试 | "你说你做过召回优化，那你优化前后的召回率指标是什么？" |

### 综合平衡型

| 场景 | 话术 |
|------|------|
| 回答模糊 | "你说的方向是对的。能再具体一点吗？比如你在这个点具体做了什么？" |
| 遇到困难 | "没问题，这个点有难度。我换一种问法……你觉得这个和 XX 有什么关系？" |
| 回答较好 | "很好，这个点理解到位了。顺着这个我想多问一层……" |
| 压力测试 | "好，那我们延伸一下：如果场景变成……你会怎么考虑？"（讨论式，不说"不够"）|

### 通用（所有风格通用）

| 场景 | 话术 |
|------|------|
| 时间紧张时 | "由于时间关系，这个问题先到这里。" |
| 连续两次答不上来 | "提示一下：这个问题涉及 XX 原理/机制。"（给出一个方向的提示，不直接给答案）|
